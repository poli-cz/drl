
# Domain Reasoning Language (DRL) 
## Neuro-Symbolic Reasoning Language for Domain Reputation Analysis

**Version:** v1.4  

DRL-Domain is a **neuro-symbolic reasoning framework** designed for explainable classification of internet domains.  
It combines **machine learning signals, symbolic reasoning rules, and LLM reasoning** into a transparent pipeline that can classify domains as:

- **Benign**
- **Phishing**
- **Malware**
- **Unknown / suspicious**

---

# Core Idea

Traditional machine learning models output only a prediction:

```
domain → phishing (0.93)
```

But they usually do **not explain the reasoning**.

DRL-Domain introduces a structured reasoning layer:

```
Features → Atoms → Rules → Decision → Explanation
```

This produces **explainable reasoning traces** 



# Architecture Overview

The DRL-Domain pipeline consists of five layers:

```
Raw Features
    ↓
Atom Engine
    ↓
Rule Engine
    ↓
LLM Reasoner
    ↓
Evaluation Engine
```

Each layer has a specific responsibility.

---

# Feature Layer

Raw attributes come from a dataset or feature extraction pipeline.

Typical attributes include:

```
rdap_domain_age
dns_MX_count
dns_NS_count
tls_has_tls
lex_name_len
ml_phish_prob
ml_malware_prob
```

Datasets are typically stored as **Parquet files**.

Example dataset row:

```
domain = login-verification.vercel.app
rdap_domain_age = 42
dns_MX_count = 0
lex_name_len = 32
ml_phish_prob = 0.91
label = phishing
```

---

# Atoms

Atoms are **atomic reasoning facts** derived from attributes.

Example:

```
rdap_domain_age = 42
→ RDAP_NEW_DOMAIN = True
```

or

```
ml_phish_prob = 0.93
→ ML_PHISH_PROB_GE_0_9 = True
```

Atoms serve as **logical evidence units** used by the rule engine.

---

## Types of Atoms

### Base Atoms

Derived deterministically from features.

Examples:

```
RDAP_NEW_DOMAIN
DNS_NO_MX
LEX_LONG_NAME
TLS_SUSPICIOUS
```

Generated by:

```
providers/base_thresholds_provider.py
```

---

### Learned Atoms

Generated from machine learning models or statistical signals.

Examples:

```
ML_PHISH_PROB_GE_0_9
ML_MALWARE_PROB_GE_0_85
CAL_RISK_GE_0_7
```

Generated by providers such as:

```
providers/ml_probs_provider.py
providers/risk_calibration_provider.py
```

These atoms represent **model evidence**, not ground truth.

---

# Atom Storage System

DRL-Domain separates atom storage into two layers.

---

## Atom Definition Store

Location:

```
atoms/atoms_registry.json
```

Defines what each atom means.

Example:

```json
{
  "name": "ML_PHISH_PROB_GE_0_9",
  "type": "bool",
  "description": "ML phishing probability >= 0.90",
  "provider_id": "ml_probs"
}
```

Purpose:

- governance
- versioning
- documentation
- validation

---

## Atom Value Store

Implemented in:

```
atom_store.py
```

Stores computed atoms per domain using **SQLite**.

Example record:

```
domain = example.com
provider = ml_probs@1.0.0
atom = ML_PHISH_PROB_GE_0_9
value = True
timestamp = ...
```

Benefits:

- avoids recomputing expensive ML signals
- enables reasoning reproducibility
- enables historical analysis

---

# Atom Providers

Atoms are computed by **provider plugins**.

Location:

```
providers/
```

Each provider implements:

```
compute(domain, attributes) → atoms
```

Providers allow new reasoning signals to be added without modifying the core system.

Examples:

```
providers/base_thresholds_provider.py
providers/ml_probs_provider.py
providers/risk_calibration_provider.py
```

---

# Atom Engine

File:

```
atom_engine.py
```

Responsibilities:

- load atom providers
- compute atoms
- validate atoms against registry
- merge atoms
- cache atoms in SQLite

Example usage:

```python
from atom_engine import AtomEngine
from atom_store import AtomValueStore

engine = AtomEngine.load_default()
store = AtomValueStore("atoms_cache.sqlite")

atoms = engine.compute_atoms(
    domain="example.com",
    attributes=row_features,
    store=store
)
```

---

# Rule Engine

File:

```
rule_engine_reference.py
```

Rules convert atoms into **evidence scores**.

Example rule:

```
IF RDAP_NEW_DOMAIN → generic_risk +3
```

Example learned-atom rule:

```
IF ML_PHISH_PROB_GE_0_9 → phishing +2
```

Rules accumulate scores:

```
benign
phishing
malware
generic_risk
```

The final classification is derived from the highest score.

---

# DRL Language

The reasoning output is written using the **Domain Reasoning Language (DRL)**.

Example structure:

```
DRL/1.2

ENTITY: domain("example.com")

ATTRIBUTES:
...

ATOMS:
...

RULES_FIRED:
...

SCORES:
...

DECISION:
...
```

This format enables **explainable reasoning logs**.

# LLM Reasoner

Location:

```
evaluation/gpt_reasoner.py
```

The LLM reasoner produces human-readable reasoning explanations.

Version 1.4 supports **Azure OpenAI deployments**.

Example configuration:

```
AZURE_OPENAI_API_KEY=KEY
AZURE_OPENAI_ENDPOINT=https://lakmoosgpt.openai.azure.com/
AZURE_OPENAI_API_VERSION=2024-12-01-preview
AZURE_OPENAI_DEPLOYMENT=Lakmoos-gpt4-o
```

The system automatically loads environment variables from `.env`.

Example usage:

```python
from evaluation.gpt_reasoner import GPTReasoner

reasoner = GPTReasoner()

reasoning = reasoner.reason(prompt)
```

# Evaluation Engine

Location:

```
evaluation/evaluate_dataset.py
```

Example run:

```
python evaluation/evaluate_dataset.py     --dataset data/domains_eval.parquet     --model gpt-4o     --output experiments/results.json
```

# Philosophy

DRL-Domain follows three design principles:

### Explainability
Every decision must be traceable.

### Modularity
New reasoning signals can be added via providers.

### Neuro-Symbolic AI
Combine machine learning predictions with symbolic reasoning.
